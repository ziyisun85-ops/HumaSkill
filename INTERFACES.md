# HumaSkill — Shared Interfaces

> **This is the binding contract for all coding agents.**  
> Every agent MUST follow these interfaces exactly.  
> No agent may change an interface without updating this document and getting approval.

---

## 1. SkillInfo

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class SkillInfo:
    name: str
    tags: list[str]
    duration_range: tuple[float, float]
    start_pose: str
    end_pose: str
    risk: str
    description: str = ""
    backend: str = "dummy"
    policy_id: str | None = None
    checkpoint: str | None = None
    action_type: str | None = None
    obs_adapter: str | None = None
```

### Field Semantics

| Field | Type | Description |
|---|---|---|
| `name` | `str` | Unique skill identifier |
| `tags` | `list[str]` | Skill tags for filtering (e.g., `["dance", "happy"]`) |
| `duration_range` | `tuple[float, float]` | Allowed duration range `(min, max)` |
| `start_pose` | `str` | Required starting pose (`"standing"`, `"low_pose"`, `"any"`) |
| `end_pose` | `str` | Expected ending pose after execution |
| `risk` | `str` | Risk level: `"low"`, `"medium"`, or `"high"` |
| `description` | `str` | Human-readable skill description |
| `backend` | `str` | Target backend name (MVP: `"dummy"`) |
| `policy_id` | `str \| None` | Policy identifier for trained skill policy (future) |
| `checkpoint` | `str \| None` | Checkpoint path (future: `.pt`, `.pth`, `.pkl`, `.npz`) |
| `action_type` | `str \| None` | Action type for simulation (future: `"joint_position"`, `"torque"`, `"velocity_command"`) |
| `obs_adapter` | `str \| None` | Observation adapter name for policy (future) |

## 2. Raw Sequence Item

Composer output item:

```python
{
    "skill": "arm_wave",
    "duration": 1.5
}
```

### Rules
- `skill` must be a string matching a registered skill name
- `duration` must be a positive number
- Raw sequences are generated by the composer
- Raw sequences pass through `SequenceValidator` before entering `TransitionManager`

---

## 3. Repaired Sequence Item

TransitionManager output item:

```python
{
    "skill": "stand_stable",
    "duration": 0.8,
    "source": "transition_inserted"
}
```

### Allowed `source` Values

| Value | Meaning |
|---|---|
| `"agent"` | Item came from composer output |
| `"transition_inserted"` | Item inserted by TransitionManager for pose bridging |
| `"recovery_inserted"` | Item inserted by execution recovery logic |
| `"duration_clamped"` | Original item had out-of-range duration, clamped to valid range |

### Requirements
Every repaired sequence item MUST contain: `skill`, `duration`, `source`

## 4. ExecutionResult

Backend MUST return a structured `ExecutionResult`. Backend MUST NOT return raw strings.

```python
from dataclasses import dataclass, field
from typing import Any


@dataclass
class ExecutionResult:
    status: str
    skill: str
    duration: float
    steps: int = 0
    reward: float | None = None
    final_obs: dict[str, Any] | None = None
    info: dict[str, Any] = field(default_factory=dict)
    failure_reason: str | None = None
```

### Allowed `status` Values
- `"success"` — skill executed successfully
- `"failed"` — skill execution failed

### Field Semantics

| Field | Type | Description |
|---|---|---|
| `status` | `str` | `"success"` or `"failed"` |
| `skill` | `str` | Skill name that was executed |
| `duration` | `float` | Planned execution duration |
| `steps` | `int` | Backend environment step count |
| `reward` | `float \| None` | Cumulative reward (future backends) |
| `final_obs` | `dict \| None` | Final observation after execution (future) |
| `info` | `dict` | Backend-specific extra information |
| `failure_reason` | `str \| None` | Reason for failure, if status is `"failed"` |

### MVP (DummyBackend) Fills
`status`, `skill`, `duration`, `steps`, `failure_reason`, `info`

### Future MuJoCo/Gym Backend Adds
`steps`, `reward`, `final_obs`, `info`, plus `terminated`, `truncated`

### Future Trained Policy Backend Embeds in `info`
`policy_id`, `checkpoint`, `obs_adapter`, `action_type`, `env_steps`

```python
{

    "source": "agent",
    "status": "success",
    "start_time": 0.0,
    "end_time": 1.0,

    "backend_reward": None,
    "failure_reason": None,
    "backend_info": {}
}
```

### Allowed `status` Values
- `"success"`
- `"failed"`

### Rules
- `start_time` and `end_time` use planned time accumulation (MVP: no real sleep needed)
- `status` comes from `ExecutionResult.status`
- `backend_info` comes from `ExecutionResult.info`

## 6. Backend Interface

```python
from abc import ABC, abstractmethod


class BaseBackend(ABC):
    @abstractmethod
    def execute(self, skill_name: str, duration: float) -> ExecutionResult:
        """Execute a skill and return a structured execution result."""
        raise NotImplementedError
```

**Every backend MUST return `ExecutionResult`.**  
**`ExecutionResult.status` MUST be `"success"` or `"failed"`.**

## 11. Transition Repair Rules

TransitionManager MUST follow these rules in order:

1. Composer output items are marked `source = "agent"`
2. If `duration` exceeds skill `duration_range`, clamp to valid range
3. If duration was clamped, mark `source = "duration_clamped"`
4. If current pose matches next skill's `start_pose`, keep the item as-is
5. If `start_pose` is `"any"`, allow starting from any pose
6. If current pose is `"low_pose"` and next skill requires `"standing"`, insert `stand_up` first
7. If current pose and next skill `start_pose` don't match, insert `stand_stable`
8. Before a `"high"` risk skill, insert `stand_stable`
9. After a `"high"` risk skill, insert `stand_stable`
10. After a `"medium"` risk skill, insert `stand_stable`
11. All inserted items MUST include a `source` field
12. All output items MUST include: `skill`, `duration`, `source`

### Default Insert Durations

| Skill | Duration |
|---|---|
| `stand_stable` | 0.8 |
| `stand_up` | 1.2 |
| `recover` | 1.5 |

## 13. General Constraints for All Implementations

- Python 3.10+
- Dependencies: PyYAML, pytest only (no other third-party packages)
- Use docstrings for core functions
- Use clear exception messages
- Handle paths that work on Windows and Linux
- Backend returns `ExecutionResult`, never raw strings
- Executor logs use `status`, never raw result strings
- Tests must not assume backend returns raw strings
